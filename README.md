## The Problem

AI coding tools need LSP servers for code intelligence ‚ÄĒ diagnostics, go-to-definition, find-references. But:

- **Servers are scattered** ‚ÄĒ npm, pip, cargo, go install, GitHub release binaries. Each has different steps on each OS.
- **Config is manual JSON editing** ‚ÄĒ Each tool stores LSP config differently. Copy-paste and pray.
- **Windows is broken** ‚ÄĒ `.cmd` wrapper spawning fails, file URIs are malformed, line endings crash servers.
- **Silent failures** ‚ÄĒ Servers fail to start with zero feedback.

## The Solution

One command. All your LSP servers. All your AI tools. All platforms.

```bash
npm install -g lspforge
```

## Quick Start

### Auto-detect everything

```bash
cd your-project
lspforge init
```

That's it. lspforge will:
1. **Scan** your project for languages (TypeScript? Python? Rust? Go?)
2. **Detect** which AI tools you have installed
3. **Install** the right LSP servers, sandboxed in `~/.lspforge/`
4. **Configure** each AI tool automatically
5. **Health check** every server with a real LSP handshake

### Install a specific server

```bash
lspforge install pyright
```

## Supported AI Tools

| Client | LSP Support | Config Written By lspforge | Post-Install Steps |
|--------|------------|----------------------------|--------------------|
| **Claude Code** | Native (plugin system) | `~/.claude/plugins/lspforge/.lsp.json` | Load plugin: `claude --plugin-dir ~/.claude/plugins/lspforge` |
| **GitHub Copilot CLI** | Native | `~/.copilot/lsp-config.json` | Restart Copilot CLI to pick up config |
| **OpenCode / Crush** | Native | `opencode.json` or `.crush.json` in project root | Restart OpenCode/Crush |

### Not supported (no LSP)

| Client | Reason | Status |
|--------|--------|--------|
| **Gemini CLI** | MCP only, no LSP | Watching ‚ÄĒ [LSP feature requested](https://github.com/google-gemini/gemini-cli/issues/2465) |
| **OpenAI Codex** | MCP only, no LSP | Watching ‚ÄĒ [LSP feature requested](https://github.com/openai/codex/issues/8745) |
| **Cline CLI** | MCP only, no LSP | Not planned |

> lspforge only integrates with tools that have **native LSP support**. We don't write LSP servers as MCP entries ‚ÄĒ they're different protocols.

## Commands

| Command | What it does |
|---------|-------------|
| `lspforge init` | Scan project, detect languages, install & configure everything |
| `lspforge install <server>` | Install a specific LSP server |
| `lspforge uninstall <server>` | Remove a server and clean up configs |
| `lspforge list` | Show installed servers (use `--available` to see registry) |
| `lspforge check` | Health check all servers (real LSP handshake) |
| `lspforge doctor` | Full environment diagnostics |

## How It Works

```
You run: lspforge install pyright

1. Registry lookup     ‚Üí Find packages/pyright/package.yaml
2. Platform detect     ‚Üí win32 x64, npm available
3. Source select       ‚Üí npm defined + npm on PATH ‚Üí use npm
4. Install             ‚Üí npm install --prefix ~/.lspforge/servers/pyright
5. Verify binary       ‚Üí Check pyright-langserver exists
6. Health check        ‚Üí Send LSP initialize, expect capabilities back
7. Config generate     ‚Üí Detect Claude Code ‚Üí write .lsp.json plugin
8. State update        ‚Üí Track in ~/.lspforge/state.json
```

All servers are **sandboxed** in `~/.lspforge/servers/` ‚ÄĒ no global pollution, no conflicts.

## Why lspforge?

### Windows-first

Most tools are built Linux/Mac-first. Windows users suffer the most:
- npm `.cmd` wrapper spawning fails ‚Üí lspforge uses shell mode automatically
- File URIs malformed (`file://C:\` vs `file:///C:/`) ‚Üí lspforge generates correct URIs
- Config lock files block onboarding ‚Üí lspforge handles this gracefully

We built on Windows. We test on Windows. It works on Windows.

## Storage

```
~/.lspforge/                     # %LOCALAPPDATA%\lspforge on Windows
  state.json                     # Tracks installed servers
  servers/
    pyright/                     # Each server gets its own directory
      node_modules/.bin/pyright-langserver
    rust-analyzer/
      rust-analyzer
    gopls/
      bin/gopls
```

## Security

- **Version pinning** ‚ÄĒ Registry entries specify exact versions
- **Sandboxed installs** ‚ÄĒ Servers can't interfere with each other or your system
- **Managed config entries** ‚ÄĒ Tagged with `_managed_by: lspforge` so we never touch your manual config
- **No auto-install without confirmation** ‚ÄĒ `lspforge init` shows you what it'll do first
